# ADR 002 — Use raw SQL files for migrations instead of a migration framework

**Status:** Accepted  
**Date:** Project inception

---

## Context

Database schema changes need to be tracked and applied consistently across all environments (local, CI, future deployments). Options include:

1. Raw numbered `.sql` files applied with `psql` in a shell loop
2. Flyway — a Java-based migration framework
3. Liquibase — another Java-based framework supporting XML/YAML/SQL changelogs
4. A Node.js migration library (e.g. `node-postgres-migrate`, `db-migrate`, `knex` migrations)

## Decision

Use raw numbered `.sql` files applied with a simple shell loop (`for f in migrations/*.sql | sort; do psql -f "$f"; done`).

## Reasons

**Zero framework dependencies.** No Java runtime, no extra Docker image, no npm package to version-pin. The only tool needed is `psql`, which is available everywhere Postgres is.

**SQL is the lingua franca.** Every database professional can read and write SQL. A migration file written in raw SQL is readable without knowing any framework. A Liquibase XML changelog is not.

**Sufficient for a PoC.** Migration frameworks add value when you need: automatic tracking of which migrations have run (via a `schema_history` table), rollback support, and team-scale coordination. For a PoC with a small team, a numeric prefix convention provides sufficient ordering and the `IF NOT EXISTS` idiom provides idempotency.

**Transparent.** The shell loop is six lines that any developer can read and understand immediately. There is no magic.

## Trade-offs

**No automatic tracking.** A migration framework records which files have run in a database table and refuses to re-run them. The raw SQL approach relies on `IF NOT EXISTS` for safety and does not prevent running a file twice. For a PoC this is acceptable.

**No built-in rollback.** Frameworks can generate down-migrations. Here, rollbacks require manually written `ALTER TABLE DROP COLUMN` or similar SQL in a new migration file.

## When to graduate

If this project evolves beyond a PoC and multiple developers are regularly writing migrations in parallel, adopt Flyway or a Node.js migration tool that tracks applied migrations in a `schema_history` table.

# ADR 003 — Use TypeScript for the application layer

**Status:** Accepted  
**Date:** Project inception

---

## Context

The application layer needs a query interface to the database. Language options include Python, TypeScript/Node.js, Go, or Java.

## Decision

Use TypeScript with Node.js.

## Reasons

**Type safety catches bugs before runtime.** TypeScript's compiler checks that query results match declared interfaces. Calling a property that doesn't exist on a DB row is a compile-time error, not a production bug.

**Same language as most web teams.** If this PoC evolves into a web API (Express, Fastify, Next.js), the query layer is already written in the right language and can be imported directly.

**The `pg` ecosystem is mature.** The `node-postgres` (`pg`) library is the most widely used Postgres client for Node.js, battle-tested, and well-documented.

**Commander.js makes great CLIs.** The PoC benefits from a usable CLI as a demonstration interface. Commander.js handles argument parsing, help generation, and subcommands with minimal boilerplate.

**Fast iteration.** `ts-node` runs TypeScript directly without a build step during development, keeping the feedback loop short.

## Trade-offs

**Node.js is not the natural home for data work.** Python's ecosystem (pandas, numpy, sklearn, matplotlib) is far richer for analytics. This is why the pipelines are Python — TypeScript handles the application/query layer, Python handles the data processing layer.

**`ts-node` is slow to start.** Each `ts-node` invocation takes 1–2 seconds to compile before running. For a CLI tool this is acceptable. For a long-running server it would be compiled first (`tsc`) and run as plain JavaScript.

# ADR 004 — Use Python for data pipelines

**Status:** Accepted  
**Date:** Project inception

---

## Context

The data ingestion pipelines need to: download files from Kaggle, read large CSVs, clean and transform data, and bulk-load into Postgres. Language options are Python, TypeScript/Node.js, or a dedicated ETL tool (dbt, Airbyte, etc.).

## Decision

Use Python with pandas and SQLAlchemy.

## Reasons

**pandas is the industry standard for tabular data work.** Reading a CSV, renaming columns, filtering bad rows, and loading to a database is ten lines of pandas. The equivalent in TypeScript requires multiple packages and significantly more code.

**Kaggle provides an official Python CLI.** The `kaggle` package handles authentication, download, and extraction. There is no official TypeScript equivalent.

**SQLAlchemy + pandas integrate natively.** `df.to_sql(engine)` bulk-loads a DataFrame into any SQL database. This pattern is well-documented, performant, and handles type mapping automatically.

**Matplotlib and Seaborn for analysis output.** Generating charts as part of the pipeline is trivial in Python. The TypeScript layer can expose the results from the database; the Python layer generates the visual artifacts.

**Separation of concerns.** Keeping pipelines in Python and the application in TypeScript means each language is used for what it does best. The database is the interface between them.

## Trade-offs

**Two languages in one repo.** Contributors need to be comfortable in both Python and TypeScript, or limit their work to one layer. For a PoC this is intentional — it demonstrates a realistic polyglot data stack.

**Python environment management.** Python version and dependency management (pip, venv, requirements.txt) is more fragile than npm. Docker mitigates this by providing a consistent Python 3.11 environment regardless of what's installed on the developer's Mac.

## Alternatives considered

**dbt for transformations:** dbt is excellent for SQL-based transformations but requires a running Postgres and doesn't handle data ingestion from external sources. It would be a good addition for a more mature version of the analytics layer.

**Airbyte for ingestion:** Airbyte is a full ingestion platform. It is far heavier than needed for a PoC and would obscure the mechanisms being demonstrated.
